# Architecture & Design Decisions

Decisions made for "The Block," the OPENLANE buyer-side auction prototype. Append-only; superseded entries are marked but not removed. See `docs/CHANGELOG.md` for shipping log and `docs/LESSONS.md` for hard-won knowledge.

Numbering: `D001` upward, append-only. Cross-link to lessons as `L###`.

---

## Setup phase (2026-05-16)

### D001: React + Vite + TypeScript for the frontend
**Date:** 2026-05-16
**Context:** Challenge permits any framework. Frontend-only is allowed.
**Decision:** React 19 + Vite + TypeScript. No backend. Data loaded statically from `data/vehicles.json` via `import` at build time.
**Rationale:** TypeScript catches a class of mistakes at compile time that would otherwise surface in review. Vite gives a fast dev loop. Static JSON import avoids a fake API layer we'd have to design and defend without adding any product value.

### D002: Tailwind v4 via `@tailwindcss/vite`
**Date:** 2026-05-16
**Context:** Need a styling approach that lets us ship polished UI without burning time on CSS architecture.
**Decision:** Tailwind v4 with the Vite plugin (`@tailwindcss/vite`). Single `@import "tailwindcss"` in `src/index.css`. No PostCSS config needed.
**Rationale:** Tailwind v4's Vite-native plugin removes the historical Tailwind setup friction. Utility-first matches the "polish + craft" rubric ŌĆö every spacing/color decision is visible in the JSX. Alternative considered: CSS modules with a hand-rolled design-token file. Slower; less defensible to reviewers who'd expect a framework choice.

### D003: localStorage for bid persistence
**Date:** 2026-05-16
**Context:** A buyer placing a bid and refreshing the page should see their bid retained. No backend is in scope.
**Decision:** `localStorage` keyed under the `openlane-block:` namespace. Bids stored as `Record<vehicleId, UserBid[]>` at `openlane-block:bids:v1`.
**Rationale:** Per-browser persistence is the right tier for a prototype ŌĆö adds zero infrastructure and demonstrates the core "place a bid, refresh, still there" user contract. Alternatives: (a) in-memory only ŌĆö refresh loses bids, which breaks the user expectation; (b) Node + JSON backend ŌĆö adds review surface and operational concerns without improving the prototype's product surface. The `v1` suffix in the key lets us migrate the schema later without colliding with old data.

### D004: Vitest + React Testing Library for tests
**Date:** 2026-05-16
**Context:** Bid validation is the only piece of substantive pure logic in the app; correctness matters and is cheap to verify. The toolchain needs to integrate with Vite without duplicating config.
**Decision:** Vitest as the runner (Vite-native, fast), RTL for any component tests. Test files co-located with source (`bidding.test.ts` next to `bidding.ts`). Setup file at `src/test/setup.ts` imports `@testing-library/jest-dom/vitest`.
**Rationale:** Vitest reuses the existing Vite config; no parallel toolchain to maintain. RTL is the React community default for component tests if we add any. Test priority is the bid validator first ŌĆö pure logic, high consequence, boundary-rich ŌĆö with component/integration tests added if time allows.

### D005: Centralized currency formatting
**Date:** 2026-05-16
**Context:** Currency is rendered across many components (cards, VDP headline, bid panel, comp panel, "your bids" list). Inlining `$` symbols and ad-hoc formatting invites inconsistency ("$22,800" vs. "$22800.00") and makes any future locale/currency change a search-and-replace risk.
**Decision:** All currency renders go through `formatCurrency(amount)` in `src/lib/format.ts`, which reads from a single `CURRENCY = { locale, code }` constant. **Never** inline `${value}` or a `$` symbol in JSX. The `Intl.NumberFormat` instance is computed once at module load.
**Rationale:** A single formatting choke-point enforces consistency across the app and reduces the locale/code to a one-line change at the constant. The architectural cost is trivial (a helper + a constant); the readability and maintainability win is durable. Caveat: the same rule applies to `formatOdometer()` if we ever need a unit change (kmŌåömi).

### D006: Time-shift normalization for `auction_start`
**Date:** 2026-05-16
**Context:** Dataset's `auction_start` values cluster around April 2026; today is May 16, 2026. Without shifting, every lot looks ended and the demo is dead.
**Decision:** `shiftAuctionStart(iso, now)` in `src/lib/time.ts` offsets every dataset timestamp by `now - DATASET_ANCHOR` where `DATASET_ANCHOR = 2026-04-05T12:00:00`. UI computes status against the shifted timestamp. Document explicitly in the README.
**Rationale:** Challenge README explicitly permits this: *"If you want to show countdowns or 'live' states, it's fine to normalize them relative to 'now' in your prototype."* Better to surface a working demo with a stated assumption than a technically-honest demo where every lot is "Ended." Alternative considered: rewrite `data/vehicles.json` timestamps ŌĆö rejected because the dataset is part of the deliverable and modifying it would look like cheating.

### D007: Match OPENLANE's actual vocabulary
**Date:** 2026-05-16
**Context:** Research of openlane.com / openlane.ca shows specific industry vocabulary: "Floor Price" (not "Reserve"), "Active/Closing/Pending" (auction state phases), "Vehicle Details Page" / "VDP," "Proxy Bid" / "Auto Bid" (max-bid mechanic). Using generic auction language would read as "didn't do the research."
**Decision:** UI labels use OPENLANE's vocabulary. The dataset field is `reserve_price`; the UI labels it "Floor Price." The detail page is the "Vehicle Details Page." Auction state pills use "Upcoming / Active / Ended." The "Closing" phase (last N minutes of a Live lot) is a stretch if time permits. **Dataset field names stay as-is** ŌĆö only the display layer aliases them.
**Rationale:** Cheap craft signal. Speaks to "judgment" criterion: chose deliberate domain vocabulary over default invention. Caveat: if a reviewer prefers the more generic "Reserve / Reserve Met," we can defend the choice with the research.

### D008: Stretch features ŌĆö comp indicator + faithful state machine
**Date:** 2026-05-16
**Context:** Challenge says stretches should make the buyer experience "clearer, more useful, or more trustworthy." OPENLANE UI research identifies "embedded comp pricing on cards" and "title-brand prominence" as their visible gaps.
**Decision:** Ship two stretches:
1. **Smart Price comp indicator** ŌĆö on each inventory card, a green/yellow/red badge comparing the current bid to a "fair price band" computed from 3 nearest comps (same make+model, year ┬▒1, similar mileage band). On the VDP, expand into a comp panel showing the 3 comps with their current bids + condition grades. (Maps to "more useful" ŌĆö answers a real buyer question.)
2. **Faithful auction state machine + title-status prominence** ŌĆö use OPENLANE phase names, "Floor Price / Floor Met" labels, live countdown on the VDP, sort-by-ending-soonest. Bundle in: salvage / rebuilt title-brand red banners on cards + VDP, default filter excludes salvage unless opted in. (Maps to "more trustworthy" and "clearer.")

Two stretches held in reserve if time runs out on these:
3. Watchlist with `/watchlist` route
4. Quick-bid increment buttons + bid-history sparkline

**Rationale:** Comp indicator on cards is a feature OPENLANE *doesn't have publicly* ŌĆö directly exploits the research gap and signals product thinking. Faithful state machine reads as someone who learned the domain. Both stretches reuse the existing types, format helpers, and bidding module, so build cost is mostly UI.
**Caveat:** If the comp algorithm is slow over 200 vehicles ├Ś3-comp lookups in render, memoize via `useMemo` keyed on filter state.

### D009: Maintain DECISIONS / LESSONS / CHANGELOG / CUTS docs through the build
**Date:** 2026-05-16
**Context:** A short-lived build benefits from a paper trail any future reader can use to understand *why* the code looks the way it does. Without it, the rationale behind non-obvious choices evaporates by the next session.
**Decision:** Keep `docs/DECISIONS.md`, `docs/LESSONS.md`, `docs/CHANGELOG.md`, and `docs/CUTS.md` updated as the build proceeds. Use the `docs-maintenance` skill workflow (copied into `.claude/skills/`). The README's final pass distills these into "Notable Decisions" and "What I'd Do With More Time" sections.
**Rationale:** Costs ~10 minutes per build session; produces a coherent record of decisions, deferrals, and learned lessons. CUTS.md in particular pays for itself ŌĆö every cut has a documented why, so deferred work isn't mistaken for missed work.
**Caveat:** Don't let doc maintenance become procrastination. If a session's decisions are trivial (e.g., picking a color), they don't earn entries.

### D010: Detailed + reflective AI-tool-use writeup in README
**Date:** 2026-05-16
**Context:** The challenge brief invites AI tool use and asks builders to be ready to explain how they used them, what decisions they made, and what they would refine. A vague answer ("I used Claude as autocomplete") is less honest than the actual workflow and harder to learn from than a specific one.
**Decision:** README dedicates a top-level section to AI tool use. Names specific files / patterns / sessions where Claude Code contributed. Includes a "what I'd refine" subsection that owns the limitations of AI-generated code.
**Rationale:** Specific and reflective beats vague and defensive. Naming the files and the contribution shape gives the reader something concrete to evaluate; the "what I'd refine" subsection signals critical engagement with the tooling rather than uncritical reliance on it.

## Build phase

### D011: Buy-now price is an inclusive upper bound for bids
**Date:** 2026-05-16
**Context:** `validateBid` rejects amounts strictly greater than `buy_now_price`. The exact-equal boundary (`amount === buy_now_price`) needed a deliberate ruling ŌĆö it could be read either way, and the test suite has to assert one.
**Decision:** Inclusive ceiling. A bid exactly equal to `buy_now_price` is valid; one dollar above is rejected with reason `above_buy_now`. The boundary is covered by an explicit test case in `bidding.test.ts`.
**Rationale:** Matches the buyer mental model ŌĆö "I'll just match the Buy Now price" should succeed, not silently fail. Inclusive is also how OPENLANE's own UI treats the threshold in practice. The alternative (exclusive) creates a confusing one-dollar gap between "highest legal bid" and "Buy Now" with no product justification.
**Caveat:** If a future requirement separates "place bid" from "take buy now" as distinct intents (so reaching the buy-now price *auto-converts* the action), this rule changes. Today they're the same submit path, so the ceiling is the right semantic.

### D012: `validateBid` is pure math ŌĆö no auction-status, no time, no DOM
**Date:** 2026-05-16
**Context:** It's tempting to fold "lot must be Active to accept bids" into `validateBid` so the validator becomes a one-stop gate. But auction status is a function of `now`, which makes the validator time-coupled and pulls a clock into the test surface.
**Decision:** `validateBid` (and the rest of `bidding.ts`) sees only `(amount, vehicle, userBids)`. Auction-status gating lives in the UI: the Place Bid button is disabled when `auctionStatus(vehicle) !== 'active'`, and submission is short-circuited before `validateBid` is even called. The bidding module imports nothing from `time.ts`.
**Rationale:** Separates the two questions a reviewer would want answered independently: (1) is this *amount* well-formed against this lot? (2) is this lot *currently accepting* bids? Mixing them couples the unit tests to a clock and makes "what does the validator do" harder to reason about. Pure-math validators also compose better with future surfaces ŌĆö e.g., a "what would this bid look like" preview tooltip that runs even on Ended lots.
**Caveat:** This shifts a responsibility onto callers (UI must gate on status). The discipline is small ŌĆö `disabled={status !== 'active'}` on the bid button ŌĆö and is enforced by the M6 smoke check.

### D013: Pin all dependencies to exact versions
**Date:** 2026-05-16
**Context:** A loose `package.json` with caret ranges (`^1.2.3`) means every reinstall can drift to a new transitive tree and silently change behavior. For a deliverable other people will clone and run, that reproducibility gap is the wrong default.
**Decision:** Pin every direct dependency to an exact version (no `^`, no `~`). Enforce going forward with `.npmrc` containing `save-exact=true`. Lock the Node major version via `.nvmrc` (currently `24`). Run `npm audit` after every install; address high/critical CVEs explicitly, never via `npm audit fix --force`. Commit `package-lock.json`.
**Rationale:** Reproducible installs are a low-cost supply-chain hygiene win. The `.npmrc` flag prevents accidental loosening when future deps get added. `npm audit` at Phase 1 reported 0 vulnerabilities ŌĆö no accepted-risk entries needed.
**Caveat:** Exact pins mean we won't auto-pick up patch upgrades. That's the point ŌĆö upgrades become an explicit, reviewable change.

### D014: Treat unzoned dataset timestamps as UTC via `parseDatasetIso`
**Date:** 2026-05-16
**Context:** `data/vehicles.json` stores `auction_start` as bare ISO strings with no timezone designator (e.g. `"2026-04-05T19:00:00"`). Per the ECMAScript spec, `new Date(iso)` parses such strings as **local time**. The time-shift anchor (`DATASET_ANCHOR_MS`) is declared in UTC via `Date.UTC(...)`. The two frames being different means `(now - anchor)` is computed in UTC, but `parse(datasetIso) + offset` lands the dataset value at a timezone-dependent moment ŌĆö `shiftAuctionStart` output and `auctionStatus` buckets drift by the viewer's tz offset (~5h in EST, ~9h in JST). The original `time.test.ts` cases passed because every input was Z-suffixed, masking the asymmetry between test inputs and production inputs.
**Decision:** Introduce a private `parseDatasetIso(iso)` helper in `src/lib/time.ts` that appends `'Z'` when no timezone designator is present (regex `/(Z|[+-]\d{2}:?\d{2})$/`), then parses. All three exported functions (`shiftAuctionStart`, `auctionStatus`, `msUntil`) route their ISO inputs through it. The anchor stays as `Date.UTC(2026, 3, 5, 12, 0, 0)`. Regression block (`parseDatasetIso convention (D014)` in `time.test.ts`) asserts parity between bare and Z-suffixed inputs against the dataset's actual shape ŌĆö and pins vitest's process timezone (`env: { TZ: 'America/Toronto' }` in `vite.config.ts`) plus a non-UTC-offset meta-guard so the parity assertions don't degrade into no-ops on a UTC CI runner.
**Rationale:** The dataset's missing-`Z` is the source-of-truth convention we have to accept (it's part of the deliverable, not something to mutate). Normalizing on the parsing side keeps the rest of the module timezone-agnostic and the anchor explicit. The regex tolerates the few zoned forms a future dataset variant might use (`Z`, `┬▒HH:MM`, `┬▒HHMM`) so the helper isn't fragile to format drift. Pinning the test TZ is the cheapest way to make the regression bug-shape independent of where it runs.
**Caveat:** If a future schema migration adds real local timezones to the dataset (e.g. dealer-local times in the seller's region), this convention is wrong and would silently mis-bucket lots. The fix is dataset-side: emit explicit `┬▒HH:MM` offsets and the helper continues to work.

### D015: Bundle the dataset via `resolveJsonModule` + narrow `include`
**Date:** 2026-05-16
**Context:** `data/vehicles.json` lives outside `src/`. The vite-default `tsconfig.app.json` has `include: ["src"]` and no `resolveJsonModule`, so a typed `import data from '../../data/vehicles.json'` doesn't compile. Alternatives considered: (a) `fetch('/vehicles.json')` at runtime, (b) an ambient `.d.ts` declaring the JSON module as `any`, (c) bundling via the tsconfig knobs.
**Decision:** Set `resolveJsonModule: true` and add the single file `"data/vehicles.json"` (not `"data/**"`) to `include`. Dataset is re-exported once from `src/data/vehicles.ts` as `readonly Vehicle[]` so the path and cast both live behind one choke-point.
**Rationale:** The dataset is build-time static and ~200 records ŌĆö bundling beats `fetch` (no async, no loading state, no network failure mode, no extra `public/` copy step). `resolveJsonModule: true` is the canonical TS lever for typed JSON. Narrowing `include` to the one JSON file (not a glob over `data/**`) keeps the type-check surface tight, leaves `scripts/generate_vehicles.mjs` untouched, and means the only path the compiler walks outside `src/` is the deliberate one.
**Caveat:** Bundling means the dataset ships inside the JS bundle once a consumer imports it (Phase 4+). For a 200-record demo the size impact is acceptable; we'll measure the actual delta against the Phase 1 baseline (~191 kB JS) at first import. If the dataset grew an order of magnitude, we'd want to revisit and lazy-load instead.

### D016: Split bid state across `bid-context.ts` (non-component) and `BidProvider.tsx` (component)
**Date:** 2026-05-16
**Context:** First draft of the bid state colocated everything in one `src/state/BidContext.tsx`: the `BidContext` object, `BidsByVehicle` type, the read hooks (`useBids`, `useBidsFor`, `usePlaceBid`), and the `BidProvider` component. ESLint's `react-refresh/only-export-components` rule fired: Vite's React Refresh HMR can only fast-refresh files that *only* export components ŌĆö a file mixing a component export with hook/type/constant exports forces a full module reload instead of preserving component state on edit. Three alternatives: (a) suppress the rule per-line, (b) keep one file and accept the HMR degradation, (c) split.
**Decision:** Split. `src/state/bid-context.ts` owns the context object, the types (`BidsByVehicle`, `BidContextValue`), the shared frozen `EMPTY_BIDS` sentinel, and the read hooks. `src/state/BidProvider.tsx` owns the `BidProvider` component and the `placeBid` action that hydrates from / writes through to localStorage. The provider imports `EMPTY_BIDS` from the context module so both halves use the same reference. Naming follows repo convention: utilities kebab-case (`bid-context.ts`), components PascalCase (`BidProvider.tsx`).
**Rationale:** HMR-preserving fast refresh is real ergonomic value across Phase 5+ ŌĆö the provider gets edited frequently during UI iteration, and a full module reload loses every component-level state along the tree. Suppressing the rule per-export trades a lint warning for the same HMR cost without flagging it to future readers. Keeping the convention also matches the broader React-context pattern where the context module is read-mostly and the provider is the only thing that changes shape over time.
**Caveat:** Two files for one logical unit is slightly more navigation cost. Mitigated by clear naming and the cross-link in each file's header comment. See **L001** for the generalizable lesson.

### D017: Default inventory filters hide salvage titles and ended lots
**Date:** 2026-05-16
**Context:** A buyer's first view of the inventory should lead with lots they can actually act on and that match the trust thesis (CLAUDE.md): title brands shouldn't be hidden, but salvage shouldn't be implicitly equal to a clean lot either. The dataset's ~140/200 lots are "Ended" against shifted timestamps and 14/200 are salvage ŌĆö without defaults the grid leads with hundreds of dead, mostly-clean cards interspersed with unmarked salvage.
**Decision:** `DEFAULT_FILTER_STATE` in `src/components/filter-rail-state.ts` ships with `titleStatuses: ['clean', 'rebuilt']` (salvage off, opt-in) and `auctionStatuses: ['upcoming', 'active']` (ended off, opt-in). Both deviations from the "show everything" stance are surfaced as removable chips via `enumerateActiveFilters` so the absence is visible and one click away from reversal. Rebuilt lots stay on by default ŌĆö they carry a banner but aren't excluded; rebuilt is a less severe signal than salvage and excluding them would over-filter.
**Rationale:** The trust thesis (un-burying title brands and price comps) extends to what loads first, not just what's visually decorated. Hiding ended is purely practical ŌĆö an "Inventory" view dominated by lots you can't bid on isn't an inventory view, it's an archive. Both defaults are reversible via the rail and visibly flagged via chips, so the discoverability cost is paid up front rather than buried.
**Caveat:** With the current 6h Active window (I2), the default view renders ~56 lots (51 upcoming + 9 active ŌłÆ salvage overlap). If Phase 9 finds that too sparse, the lever is `ACTIVE_WINDOW_MS` in `time.ts`, not the default ŌĆö relaxing the default to include ended would defeat the practical motivation.

### D018: Inventory filter state lives in component state, not URL params
**Date:** 2026-05-16
**Context:** Filter state needs a home. Two reasonable choices: (a) `useState` in the Inventory page, or (b) `useSearchParams` so the URL is shareable / bookmarkable / back-button-friendly. URL-state is the higher-leverage option for a production app ŌĆö shared deep links into specific filter slices are real buyer value ŌĆö but it costs a meaningful chunk of build time on serialization, parse-validation, default-pruning to keep the URL clean, and edge cases (back-button replay, query-string collisions with future routes). Issue **I4** in PLAN.md flagged this.
**Decision:** Component state via `useState` in `src/pages/Inventory.tsx`. URL-state filters are logged in `docs/CUTS.md` as a `time` cut. The `FilterState` shape and its applier live in `src/components/filter-rail-state.ts` as plain values, so a future swap to URL-state would be a localized refactor (replace the `useState` with a `useSearchParams` codec).
**Rationale:** The prototype's hour budget is better spent on the trust-thesis features (Smart Price stretch, title-brand prominence) than on URL plumbing. Component state covers every demo path and the M5 milestone exactly ŌĆö what it doesn't cover (shareable links, back-button history) isn't on the walkthrough's critical path.
**Caveat:** Refresh loses filter state. Acceptable for the prototype; would be visibly bad for a real buyer workflow. The refactor path is annotated in the doc comment at the top of `filter-rail-state.ts`.

### D019: `placeBid` reads and writes through a ref, not a closure or functional setter
**Date:** 2026-05-16
**Context:** Phase 4's `BidProvider.placeBid` was a `useCallback` that closed over `bidsByVehicle`. Two synchronous calls to `placeBid` in the same render tick (rapid double-click on Place Bid, an immediate Buy Now after a Place Bid, or any future programmatic invocation) both read the same closure: `existing` is the same array on both calls, the second `next` overwrites the first, and the first bid disappears. The standard fixes are (a) the functional setter pattern `setBidsByVehicle(prev => ...)`, or (b) a `useRef` mirror. (a) re-runs the updater under React Strict Mode, which would double-fire `writeJSON` and validate the same bid twice ŌĆö fine for pure math but a real footgun the moment side effects move inside.
**Decision:** Hold the latest state in `latestRef = useRef<BidsByVehicle>(bidsByVehicle)`. Inside `placeBid`, read `latestRef.current` for both the existing-bid array and the spread base; update `latestRef.current = next` inline with `setBidsByVehicle(next)`. The callback's dependency array empties out (no longer reads `bidsByVehicle`), so `placeBid`'s referential identity is now stable across renders too. Signature unchanged; callers untouched.
**Rationale:** The ref is synchronous ŌĆö the second `placeBid` in the same tick sees the first call's commit. Side effects stay inside the callback, where React Strict Mode runs them exactly once. The stable identity is a free bonus: any future `useEffect` or `useMemo` that depends on `placeBid` won't churn on every bid commit.
**Caveat:** A ref mirror has to be updated *every* time `setBidsByVehicle` is called, or the two states drift. Today the provider has exactly one commit site (`placeBid`); future mutators (clear-bids, undo-last) must update the ref alongside the setter. Documented in the file header so the discipline is visible at the import site.

### D020: Smart Price verdict ŌĆö inclusive band boundaries, drop unbid comps, whole-map memo key
**Date:** 2026-05-16
**Context:** Four design micro-decisions inside `src/lib/comps.ts` + the surfaces that consume it. Each is small, but each has a defensible alternative, so they're recorded together. (1) The verdict's relationship to the band edges ŌĆö is "matching the cheapest comp" fair or below? (2) Some comps have `current_bid === 0` (just-listed lots with no activity); including them pins `low` at 0 and makes "fair" cover anything between zero and the highest comp ŌĆö useless signal. (3) `'unknown'` verdicts ŌĆö render a grey "No comps" pill, or hide the badge entirely? (4) `compPriceBand` is called from `VehicleCard`, `BidPanel`, and `CompPanel`; the memo key could be just the target's bids + its comps' bids, or the whole `bidsByVehicle` map.
**Decision:** (1) Verdict boundaries are inclusive: `price < low ŌåÆ 'below'`, `low Ōēż price Ōēż high ŌåÆ 'fair'`, `price > high ŌåÆ 'above'`. (2) Drop comps with `displayedCurrentBid Ōēż 0` from the band; `compPriceBand` returns `null` when no priced comps remain. The `comps` array surfaced to the panel reflects the filter, so the VDP comp panel doesn't show a "$0" mini-card next to real ones. Floor prices were considered as a fallback but D007 forbids surfacing the reserve number; using it in the band would leak that signal. (3) `SmartPriceBadge` returns `null` when verdict is `'unknown'` ŌĆö absence of badge is more honest than a "No comps" label, and the typed `Record<ScoredVerdict, ...>` lookup table makes the union enforce "we only style rendered states." (4) Every consumer memoizes on `[vehicle, bidsByVehicle]` ŌĆö the whole map, not a precomputed subset.
**Rationale:** (1) Inclusive matches the buyer's mental model and mirrors D011's buy-now ceiling ŌĆö exact-equal boundaries should succeed at face value, not feel "almost." (2) An unbid comp tells us what the seller is *asking* (via `starting_bid`), not what the market is *clearing at*. Mixing those two signals produces a band that's wide where it shouldn't be. Filtering keeps the band reflective of actual auction activity; if filtering empties the set, the badge disappears rather than fabricating signal. (3) Same trust thesis: don't say something when there's nothing to say. The grey pill was tested and read as visual debris on the long-tail lots. (4) Computing the precise dependency subset (target's own bids + each comp's bids) requires running `findComps` *outside* the memo to identify comps before keying ŌĆö which defeats the memo. Keying on the whole map costs a referential identity check; the band recomputes only when `BidProvider` actually commits a bid (D019's ref-mirror makes commits atomic), so unrelated mutations don't churn.
**Caveat:** The whole-map key means *any* bid commit re-runs comps for every visible card on the inventory grid. With 200 vehicles ├Ś 3-comp lookups ├Ś O(pool) filter, that's still well under a frame on the M7 verification hardware, but if the dataset grew an order of magnitude the right move would be precomputing comp-id lists at module load and keying card memos on a slice of the map. Filtering unbid comps also means a make/model with few priced peers may show no badge at all ŌĆö acceptable: hiding is honest when the signal is thin.

### D021: Faithful state machine ŌĆö `auctionPhase` derived from `auctionStatus`, countdown hook on VDP only
**Date:** 2026-05-16
**Context:** Stretch B adds three surfaces: a live countdown on the VDP, a fourth "Closing" auction-status pill in the last 10 minutes of the Active window, and beefed-up title-brand prominence. Four design questions fell out as I built it. (1) Does Closing replace Active in the engine union (`AuctionStatus`) or sit beside it as a display-only state? (2) Where does the countdown's tick live ŌĆö per card on the inventory grid, or on the VDP only? (3) How does `useCountdown` avoid the `react-hooks/set-state-in-effect` lint hit that the Phase 6 `ImageGallery` ran into? (4) The plan called for a hover tooltip on the "Hide salvage" filter toggle ŌĆö but hover-only fails the accessibility floor and is invisible to keyboard users; what replaces it?
**Decision:** (1) Add `AuctionPhase = AuctionStatus | 'closing'` and a derived `auctionPhase(iso, now)` helper. `auctionStatus` stays the three-state engine that filter / sort math reads; `auctionPhase` is the display-layer promoter that flips Active ŌåÆ Closing once `msUntil(end) Ōēż CLOSING_WINDOW_MS` (10m). `BidPanel.isActive` keeps reading the engine (Closing lots still accept bids) while pill copy reads the phase. (2) `useCountdown` is used on the VDP only ŌĆö the inventory grid evaluates `auctionPhase` once per render and accepts that a card sitting on screen won't auto-flip without a refresh. Alternative (per-card tick) costs 200 timers for a signal that doesn't change the demo's load-bearing path. (3) `useCountdown` derives ms-remaining on read from `Date.now()` rather than holding it in state; the effect only increments an unused tick counter from inside the interval callback. No synchronous `setState` in an effect body ŌĆö same fix shape as Phase 6's `ImageGallery` (derived `safeIndex`). (4) Replaced the hover tooltip with an always-visible help paragraph under the Title-status section: "Salvage lots are hidden by default. These vehicles carry a salvage title brand ŌĆö warrant extra inspection and may not be financeable."
**Rationale:** (1) Engine vs. display separation mirrors D012 (pure-math validator) and D007 (vocabulary aliases live in the display layer). A four-state engine union would force every filter/sort path to enumerate Closing and likely default to "Active includes Closing" anyway. (2) The VDP is where buyers spend the seconds-to-close minutes; the grid is where they scan, not where they watch the clock. Memoryless per-card pills are honest about that ŌĆö a grid that pretends to be live is the worse lie. (3) Render-time derivation is the React-idiomatic shape and the rule-of-thumb fix the lint rule's docs point at. The unused tick counter is the conventional re-render escape hatch. (4) The trust thesis (CLAUDE.md) is that the salvage exclusion is a *visible* default, not a hidden one ŌĆö a hover tooltip undermines that. Inline copy costs three lines of vertical space and serves keyboard and first-time visitors equally.
**Caveat:** Per-second ticks on the VDP wake a re-render of the whole BidPanel subtree every second. That's fine today (the panel is small), but if it grew (a live bid stream, an order book), the right move would be to scope the tick to just the countdown line via a child component so the rest of the panel doesn't recompute on every second. The grid's static phase also means the M8 spot-check ("a lot ending within 10 min reads Closing") needs to land on a fresh navigation ŌĆö covered explicitly in the milestone verification.

### D022: Persist the session anchor to localStorage with a 24h TTL
**Date:** 2026-05-16
**Context:** After **L002** froze the shift baseline at module load (`SESSION_NOW_MS = Date.now()`), countdowns ticked correctly within a session, but a full page reload recaptured a fresh baseline ŌĆö a VDP that read "Closes in 1m 30s" before reload jumped back to its initial value after, because the shift re-anchored to the new wall clock. The fix space had several knobs that each warranted a deliberate call rather than a default. (1) Persist the anchor or accept the reload-resets-clock behavior? (2) `localStorage` (survives tab close + browser restart) or `sessionStorage` (per-tab, dies on close)? (3) TTL ŌĆö none, 1h, 24h, 7d, or longer? (4) Validation depth on the stored value? (5) Where does the side effect live ŌĆö eagerly at module load, or lazily inside the first call?
**Decision:** (1) Persist. (2) `localStorage` via the existing `readJSON` / `writeJSON` helpers under the namespaced key `session-anchor:v1`. (3) **24h TTL**: re-anchor on read if `(now - storedAt) Ōēź 24h` or `storedAt > now`. (4) Defensive read ŌĆö must be the expected `{capturedAt: number}` shape, the number must be finite, and `storedAt Ōēż now` (rejects clock-skewed / corrupted values that would push the baseline into the future). (5) Eager module-load side effect: `const SESSION_NOW_MS = resolveSessionAnchor()` at the top of `time.ts`, which writes on first import if no stored anchor is present.
**Rationale:** (1) Without persistence, every reload is a UX cliff ŌĆö the countdown the buyer was watching jumps back, which reads as either a bug or a sign the timer is fake. Persisting closes the perception gap. (2) `localStorage` matches the rest of the persistence layer (bids live in `openlane-block:bids:v1`); using `sessionStorage` would mean two different storage tiers in the same module-load lifecycle, and tab-close shouldn't reset the demo clock any more than it resets the user's bids. (3) 24h was picked specifically ŌĆö short enough that "come back next week" gets a fresh distribution rather than a fully-Ended grid (the active window is 6h, so by 24h everything that was Active at session start has long ended), long enough that "come back after lunch / overnight" feels continuous. 1h would mis-fire on common usage patterns (going to a meeting); 7d would mean a stale demo greets a returning viewer with a blank screen. (4) The shape check protects against schema drift (a future change to the stored shape gracefully re-anchors) and against the `storedAt > now` case that would set the baseline in the future ŌĆö at best the countdown sits at constants, at worst `auctionPhase` reports nonsense. The cost is three lines; the failure mode it prevents is silent. (5) Eager is correct because every other call into `time.ts` reads `SESSION_NOW_MS`; deferring to lazy would either need a getter (which loses the const-shape) or a memoization wrapper (which adds a hot-path branch for no gain). The write-on-first-load is acceptable for a client-only prototype with no SSR; the `getStorage()` guard in `storage.ts` already no-ops on non-browser environments.
**Caveat:** Cross-tab behavior is intentional but worth flagging: two tabs opened against this app share the same anchor, so a countdown in tab A and tab B will agree to within the read jitter. Tab A clearing localStorage (via DevTools or a hypothetical "reset demo" button) won't re-anchor tab B until tab B reloads ŌĆö there's no `storage` event listener wired up. Adequate for the prototype; would matter for a multi-tab production flow. The TTL also means a demo viewer who pauses on a lot for >24h and then returns will see the lot snap to "Ended" alongside a re-anchored distribution ŌĆö uncommon enough not to design around, but a known sharp edge.
